# V7 → V8 → V9 重建升級路徑：實作清單與紀錄

> 目的：建立一條「型別穩定、責任分層清楚」的新主線。
> 原則：保留舊 V7/V8 作為歷史版本；新主線從新 V7 開始，逐版升級到新 V9。

---

## 0) 核心設計決策（先固定）

1. `userId` 從新 V7 起一律使用 `string`。
2. `userId` 視為 opaque string；`"0001"` 是示例，不可在程式中假設它一定可轉數字。
3. `Auth` 與 `Store` 分層：
   - `Auth`：登入、登出、session、current user
   - `Store`：menu/orders/order_items 業務資料存取
4. `Route` 只做協調，不直接承擔認證細節。
5. 同一個 Neon DB，但以 PostgreSQL schema 隔離版本（`v8`、`v9`）。

---

## 1) 分支策略

- 歷史分支保留不動：
  - `feat/v7-*`（現有）
  - `feat/v8-*`（現有）
- 新主線分支：
  1. `feat/v7-clean-auth-store`
  2. `feat/v8-clean-auth-store`（從 1 切出）
  3. `feat/v9-clean-better-auth`（從 2 切出）

---

## 2) 實作總順序（高層）

1. 新 V7：先把分層與型別地基打好。
2. 新 V8：導入 DB，但完全沿用新 V7 的分層與型別。
3. 新 V9：接入 Better Auth，不再發生 `userId` 型別切換。

---

## 3) 新 V7 實作清單（分層地基）

### A. 契約與型別

- [ ] `shared/contracts.ts`：`User.id`、`Order.userId` 固定為 `string`
- [ ] 補註解：`userId` 是 opaque string，不可做數值運算

### B. Auth 抽象與實作

- [ ] 建立 `Auth` 介面（建議）
  - `login(credentials)`
  - `logout(session)`
  - `getSessionUser(request)`
- [ ] 建立 V7 的 `DemoAuth`（json/file based）
- [ ] 把原本在 Store 的 `login/getUserById/getUserByEmail` 移出到 Auth

### C. Store 介面與實作

- [ ] `Store` 只保留業務存取（menu/orders/order_items）
- [ ] 所有與 user 相關參數統一收 `userId: string`
- [ ] JsonFileStore 以 `string userId` 正常運作

### D. Route 層重接

- [ ] route 改為：先呼叫 `Auth.getSessionUser()`，再把 `sessionUser.id` 傳給 Store
- [ ] 所有 owner 比對改為 string 比對

### E. 驗收條件

- [ ] 不存在 `Number(userId)` / `parseInt(userId)` 這類轉換
- [ ] Store 中不再出現登入責任
- [ ] 主要 API 行為與 V7 一致

---

## 4) 新 V8 實作清單（DB 化但分層不變）

> 起點：從 `feat/v7-clean-auth-store` 切出。

### A. DB schema 命名空間

- [ ] 建立/使用 PostgreSQL schema：`v8`
- [ ] V8 業務表放在 `v8` schema
- [ ] migration 使用 V8 專用目錄與狀態管理

### B. Store 實作替換

- [ ] 將 V7 的 JsonFileStore 替換為 PgStore（或並存切換）
- [ ] 介面保持不變（沿用新 V7 Store 介面）
- [ ] `userId` 在 DB 欄位保持 `text`

### C. Auth 持續分離

- [ ] V8 仍沿用 `Auth` 抽象（DemoAuth 或 DB-backed Auth）
- [ ] 不把 auth 方法塞回 Store

### D. 驗收條件

- [ ] route 層無需修改業務流程即可從 JsonFileStore 切到 PgStore
- [ ] `userId` 仍全域 string
- [ ] V8 在 `v8` schema 可獨立 migrate/run

---

## 5) 新 V9 實作清單（Better Auth 正式接入）

> 起點：從 `feat/v8-clean-auth-store` 切出。

### A. DB schema 命名空間

- [ ] 建立/使用 PostgreSQL schema：`v9`
- [ ] Better Auth 表（user/account/session）與 V9 業務表都在 `v9`
- [ ] migration 使用 V9 專用目錄與狀態管理

### B. Auth 實作替換

- [ ] `Auth` 抽象的實作改成 Better Auth adapter
- [ ] route 層仍呼叫同一組 Auth 介面
- [ ] `session.user.id` 直接作為 `Store` 的 `userId`

### C. Store 與業務

- [ ] Store 仍不含登入責任
- [ ] owner 檢查為 string 比對
- [ ] 不存在過渡橋接型別（LegacyUser/LegacyOrder）

### D. 驗收條件

- [ ] 無 `userId` 型別切換（始終 string）
- [ ] 無 email→lookup→number 這種橋接流程
- [ ] V9 在 `v9` schema 可獨立 migrate/run

---

## 6) 同一 DB 多版本隔離規則

1. 共用同一個 `DATABASE_URL`，但分 schema：`v8`、`v9`。
2. 各版本 migration 不共用狀態紀錄。
3. 禁止跨版本 schema 的 FK 與 join。
4. 測試/示範時明確標註目前版本使用的 schema。

---

## 8) 第一輪執行建議（我們接下來要做的）

1. 先建立 `feat/v7-clean-auth-store`。

## 9) 本清單的「完成定義」

- 新 V7、新 V8、新 V9 三條分支都能獨立運行。
- 三個版本都維持 `userId: string`。
- Auth/Store 分層一致，不再回退混層。
- 教學可清楚展示「歷史版本」與「重建後主線」兩套路徑。
